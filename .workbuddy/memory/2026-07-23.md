## CI 编译失败修复：ObackAppListController performSelector -Werror
- GitHub Actions（macOS theos + roothide）编译 `ObackSettings` bundle 时报 3 个 error，全部是 `-Warc-performSelector-leaks` 被当 `-Werror` 处理：
  - `ObackAppListController.m:27` `[proxyCls performSelector:appSel withObject:bid]`
  - `ObackAppListController.m:123` `[wsCls performSelector:defSel]`
  - `ObackAppListController.m:127` `[ws performSelector:allSel]`
  - 其余 4 处（`iconSel`/`typeSel`/`bidSel`/`nameSel`）早已用 `#pragma clang diagnostic push/ignored "-Warc-performSelector-leaks"/pop` 包裹，故未报错——唯独这三处裸调用触发。
- 根因：tweak 调私有 API（`LSApplicationProxy`/`LSApplicationWorkspace`）只能用 `NSSelectorFromString` + `performSelector` 动态发消息，ARC 无法推断返回值所有权，默认开 `-Warc-performSelector-leaks`；CI 的 theos 默认开启 `-Werror`，于是编译失败。
- 修复：给这三处也补上和别处一致的 pragma 包裹（`push → ignored → 调用 → pop`）。验证：grep 确认 7 处 performSelector 全部被 pragma 覆盖，push/pop 配对 6/6（bidSel/nameSel 同块共享）。未改 Makefile / 未引入全局 `-Wno-arc-performSelector-leaks`，局部抑制最安全。
- 推送：commit `4cb5b8d`（"fix: ObackAppListController 三处裸 performSelector 补 pragma 抑制 -Warc-performSelector-leaks"）→ `aacf7d8..4cb5b8d main -> main`，1 file changed (+9)。待 CI 重跑通过。
- 经验（可复用）：**越狱 theos tweak 调私有 API 时，所有 `performSelector` / `objc_msgSend` 动态调用必须统一用 pragma 抑制 `-Warc-performSelector-leaks`，否则 CI 的 `-Werror` 会编译失败**。本工程图标 data 已用 `objc_msgSend` 强转（无需 pragma），但返回 id 的 selector 用 performSelector 时必包 pragma。

## control 版本号自动带 commit 短哈希（commit dc6842d 失败 → 2e74dec 成功）

## CI 编译失败→修复：bc0e014 的 __weak 在 MRC 下报错（commit 5acd91a）
- 用户发来 CI 失败日志：`ObackManager.m:253:9: error: cannot create __weak reference in file using manual reference counting`。根因：tweak 整个工程是**手动引用计数（MRC）**，不支持 `__weak`——我在 bc0e014 给 modal dismiss 的 completion block 里写了 `__weak UIViewController *weakTop = top;` 防循环引用，但 MRC 文件里 `__weak` 直接编译失败。
- **MRC 关键认知**：tweak 入口（`.xm`/`.m`，未开 `-fobjc-arc`）默认 MRC。`__weak` 在 MRC 下**直接编译错误**；`@property (weak)` 在 MRC 合成时也会报错。本工程 `.h` 的 property 已统一用 `assign`（如 `ObackTransitioningDelegate.original` = assign），安全。
- 修复（5acd91a）：删掉 `__weak` 声明，completion block 里直接用强引用 `top`。理由：block 由 `dismissViewControllerAnimated:completion:` 短暂持有，`top`（正在 dismiss 的 VC）不反向持有该 block，**无循环引用风险**；block 跑完即释放，不泄漏。等价替换为普通强引用即可。
- **修正 bc0e014 记忆里的描述**：原记"weak 防 dangling"有误——实际就是 `__weak` 导致编译失败，不是什么安全写法；MRC 下应直接用强引用（只要确认无反向持有）。
- 推送：`bc0e014..5acd91a main -> main`（复用用户 PAT 注入）。CI 重编，版本 `0.1.0+5acd91a`。待真机验证（本轮根因：黑屏/无法操作，已靠 dimView 关拦截 + modal 转场 interacting 闸门 + 诊断日志加固）。
- **硬规则（写入项目 MEMORY.md）**：本工程 tweak 部分是 **MRC**，所有 `.m`/`.xm` 代码**禁止使用 `__weak`**；block 里引用对象若担心循环引用，改**强引用 + 确保无反向持有**（或 MRC 下用 `__block` 不 retain），切勿写 `__weak`。`.h` 的 property 也别用 `weak`，统一 `assign`/`retain`。CI 默认 `-Werror`，任何 MRC 违例都会编译失败不出包。
